The Five Integration Options

A. Wix Headless (Full Custom Frontend) 3-4 Weeks

Deploy your custom frontend (React/Next.js) on Vercel or Netlify. Use Wix purely as a backend via their JavaScript SDK and REST APIs. No Wix pages are rendered at all.

What This Means in Practice

You'd need to convert your vanilla HTML/JS codebase into a React or Next.js app, then integrate the Wix SDK (@wix/sdk, @wix/blog, @wix/crm, etc.) for every backend feature. Blog posts, contact forms, CRM data... all fetched via API calls and rendered in your custom UI.

Pros
  • 100% control over frontend and animations
  • Best possible performance (no Wix rendering overhead)
  • Full GSAP + Lenis support without restrictions
  • Modern deployment (Vercel, Netlify, edge functions)
  • No separate pricing tier for headless
  • Official Next.js templates available as starting points
Cons
  • Requires converting vanilla JS to React/Next.js first
  • Email campaigns still need Wix dashboard for creation
  • You build and maintain ALL frontend UI (blog layout, forms, etc.)
  • Wix analytics/tracking partially lost on non-Wix pages
  • OAuth setup and session management complexity
  • 3-4 weeks minimum, not feasible in 2 weeks
B. Wix Studio Native Rebuild 4-6 Weeks

Rebuild the entire site inside Wix Studio using their native animation tools (Inspector panel, scroll triggers, parallax). Use Velo for custom JavaScript where Studio's built-in animations fall short.

What This Means in Practice

You'd recreate each section in Wix Studio's visual editor, using their timeline-based animation inspector for scroll effects. Complex GSAP animations (clip-path reveals, horizontal scroll pinning, noise-to-clarity) would need to be either simplified or embedded as HTML iframes within Wix pages.

Pros
  • Full, seamless Wix integration (CRM, email, blog, analytics, everything)
  • Visual editor for non-dev content updates
  • Wix hosting, SSL, CDN all managed
  • Studio has decent scroll animations and parallax built in
  • Velo supports custom JS for dynamic behavior
Cons
  • Cannot match your current animation quality: no GSAP at page level, no Lenis, no clip-path reveals, no horizontal scroll pinning
  • GSAP only works inside HTML embed iframes (sandboxed, isolated)
  • No direct DOM access for page-wide animations
  • Manual rebuild of every section from scratch
  • Cannot import HTML/CSS/JS into Studio
  • SVG path animation, particle systems, generative effects impossible
  • 4-6 weeks realistically, significant animation downgrade
The animation gap is real. Wix Studio's Inspector can handle entrance animations, basic parallax, and simple scroll triggers. But your codebase uses advanced GSAP features (ScrollTrigger scrub, pin, clip-path, staggered timelines) that have no native equivalent in Studio. You'd lose 60-70% of the animation sophistication.
C. Hybrid: Reverse Proxy / Path-Based Routing 10-14 Days Recommended

Serve your custom animated pages (homepage, about, services) from your own hosting. Route specific paths (/blog, /contact, /faq) to Wix. All under one domain using Cloudflare Workers, Vercel rewrites, or Nginx reverse proxy.

What This Means in Practice

A user visits florianp.com and sees your custom GSAP-animated homepage. They click "Blog" and get routed to a Wix-rendered blog page, all on the same domain. The routing layer (Cloudflare Worker or Vercel middleware) decides which server handles each request based on the URL path. Forms on your custom pages submit to Wix CRM via API. Wix email marketing, automations, and tracking continue working on the Wix-served pages.

Pros
  • Keep 100% of your GSAP animations on marketing pages
  • Wix CRM, email, blog, analytics fully intact on Wix pages
  • Single domain (best for SEO, no link equity dilution)
  • Feels like one seamless website to visitors
  • No framework conversion needed (your vanilla HTML works as-is)
  • Cloudflare Workers: free tier or ~$5/month
  • Achievable in 10-14 days with focused effort
  • Can progressively migrate more pages to custom over time
Cons
  • Styling consistency between custom + Wix pages needs work
  • Session/cookie sharing across the two backends requires setup
  • DNS propagation needs to happen in week 1 (24-48 hours)
  • Two systems to maintain going forward
  • Wix analytics only tracks Wix-served pages natively
  • Need to match header/footer/navigation across both

Implementation Timeline

Days 6-8: Integration Style the Wix blog/FAQ to match your custom pages (fonts, colors, header, footer). Set up Wix Chat widget embed on custom pages. Configure analytics (GA4 or similar) across both backends.
Days 9-10: Polish Cross-browser and mobile testing. Page transition consistency. Loading states. Error handling. Performance optimization.
Days 11-14: Buffer Bug fixes, edge cases, final QA. Soft launch, monitor, iterate.
D. Subdomain Split 2-3 Weeks

Custom site at florianp.com, Wix blog at blog.florianp.com (or vice versa). Each runs independently on its own hosting.

Pros
  • Clean separation, simple to set up
  • No reverse proxy complexity
  • Each system fully independent
  • Easy to maintain long-term
Cons
  • SEO link equity split across subdomains
  • Feels like two separate sites to visitors
  • Cookie/session sharing difficult across subdomains
  • Navigation between sites feels disjointed
  • Analytics fragmented
E. Wix with Custom Element Embeds 2-3 Weeks

Keep Wix as the primary platform. Embed your animated sections as Custom Elements (web components) inside Wix Studio pages. Animations run inside sandboxed iframes.

Pros
  • Wix remains the single platform
  • All Wix services fully functional
  • Can use your GSAP code inside embeds
  • Gradual migration possible (section by section)
Cons
  • Animations sandboxed in iframes (no full-page scroll effects)
  • Lenis smooth scroll impossible at page level
  • Performance overhead from multiple iframes
  • Horizontal scroll pinning, clip-path reveals won't work across page
  • Feels like "patches" rather than a cohesive experience
  • Mobile responsiveness challenges with iframe sizing

Analytics & Tracking: The Full Picture

This was your main concern, so let's be thorough. The good news: because everything is on the same domain (florianp.com), most tracking works seamlessly. Here's the breakdown:

GA4 (Google Analytics 4): Works Across Both Backends

GA4 uses first-party cookies tied to your domain. Since both your custom pages and Wix pages live on florianp.com, GA4 sees them as one site. A visitor who lands on your animated homepage and then clicks to the Wix blog registers as a single continuous session. No special cross-domain setup needed.

How to install: Use the same GA4 measurement ID (G-XXXXXXXXXX) everywhere. On your custom HTML pages, paste the GA4 tag in the <head>. On Wix pages, enter your G- ID in Wix's native Google Analytics integration (Settings > Marketing Integrations). Do NOT also add GTM on Wix pages if you're using the native integration... that creates duplicate events.

Wix Analytics: Only Tracks Wix Pages

This is a hard limitation. Wix Analytics can only see pages that Wix renders. Your custom animated homepage, about page, and services page will not appear in Wix Analytics reports at all. This means Wix Analytics becomes a partial view of your traffic (blog + FAQ only), not the full picture.

The solution: Use GA4 as your primary analytics tool for site-wide data. Use Wix Analytics only for blog-specific insights (it's still useful for seeing which posts perform, comment engagement, etc.). Don't try to make Wix Analytics cover the whole site... it can't in this architecture.

CRM & Lead Attribution

Here's the flow that matters: someone visits your custom homepage, browses around, then fills out a contact form on a Wix page. Does the CRM capture the full journey?

Partially. Wix CRM will capture the form submission and any Wix-page visits. It won't automatically know they visited your custom pages first. To solve this:

  • UTM parameters persist naturally across both backends on the same domain. If someone arrives via a LinkedIn ad with ?utm_source=linkedin, that parameter carries through as they navigate to a Wix form page.
  • Don't add UTM parameters to internal links. Only use them for external traffic sources. Internal navigation should use clean URLs.
  • For full journey tracking, use Google Tag Manager (GTM) as your central tag manager across both backends. GTM can track page views, scroll depth, and click events on custom pages, then pass that data to GA4 where you can see the complete user journey.

Retargeting Pixels (Meta, LinkedIn, etc.)

Same-domain means same cookies. A Meta Pixel firing on your custom homepage and a Meta Pixel firing on a Wix blog page will recognize the visitor as the same person. Install the pixel manually on your custom pages (in the <head>) and use Wix's native Meta integration for Wix pages. Use the same Pixel ID on both.

FAQ: Move It to Custom HTML

This is a strong recommendation based on the SEO/GEO research. Your 29 FAQ entries should move from Wix to a custom-built FAQ page. Here's why:

Performance

Your custom pages will load in ~1.5s (static HTML, minimal overhead). Wix FAQ pages typically load in ~3.5s due to Wix's rendering pipeline. That 2-second difference matters for both user experience and Core Web Vitals scores, which directly affect Google rankings.

Schema Markup Control

On a custom FAQ page, you have full control over FAQPage structured data (JSON-LD). You can mark up every question-answer pair exactly to Google's spec, add nested schema (mentions, citations, author), and optimize for rich snippets. On Wix, structured data options are more limited and template-driven.

GEO (AI Search) Impact

This is the big one. AI crawlers (Google AI Overviews, ChatGPT search, Perplexity) don't execute JavaScript well. Your custom HTML pages are 100% visible to them. Wix pages, which rely on client-side rendering, are only about 70-80% visible. For FAQ content... which is exactly the kind of Q&A format AI models love to cite... being on a fast, fully crawlable custom page gives you significantly higher chances of appearing in AI-generated answers.

GEO citation rates: Research shows custom static pages get cited in AI search results 40-50% of the time when relevant. Wix-rendered pages with similar content sit around 25-35%. For FAQ content specifically, that gap is your competitive advantage.

Implementation

Export your 29 FAQ entries from the Wix FAQ Database. Build a custom FAQ page with accordion-style UI (GSAP animations for open/close if you want). Add comprehensive FAQPage JSON-LD schema. Set up a 301 redirect from the old Wix FAQ URL to the new custom one. This is a 1-2 day task, well within the two-week timeline.

SEO/GEO Deep Dive

The Good News

Google treats all pages on the same domain equally, regardless of which backend serves them. A reverse proxy is a completely legitimate architecture. There are no crawling or indexing penalties for routing different paths to different servers. Internal links between custom and Wix pages pass link equity normally. You keep a single domain, which means zero SEO dilution.

The Watch-Outs

Core Web Vitals gap. Your custom static pages will score 90-100 on Google's performance metrics. Wix pages typically score 30-50. Google evaluates Core Web Vitals at the page level, not the site level, so your custom pages will rank well independently. But the Wix pages (blog, contact) will need aggressive optimization: compress images, minimize custom code, use Wix's built-in lazy loading. This gap is not a dealbreaker, but it's something to actively manage.

Unified sitemap. You'll need to generate a combined sitemap.xml that includes URLs from both backends. Your custom pages won't auto-generate a sitemap, so you'll create one manually (it's just an XML file listing your URLs). Wix generates its own. Merge them and submit the combined version to Google Search Console.

robots.txt must come from one place. Since robots.txt lives at the root domain, your reverse proxy needs to serve a single robots.txt that covers both backends. Make sure it doesn't accidentally block Wix paths or vice versa.

GEO Strategy

Generative Engine Optimization is where this hybrid approach actually shines compared to an all-Wix setup:

  • Your custom pages are fully readable by AI crawlers. Static HTML with clean semantic markup is exactly what GPTBot, ClaudeBot, and PerplexityBot prefer. No JavaScript rendering issues.
  • Entity signals matter. Make sure your custom pages include Organization schema (JSON-LD) with your name, business description, and sameAs links to LinkedIn, social profiles, etc. This helps AI models build a strong entity profile for "Florian Philippe" and "Brand Therapy."
  • FAQ on custom HTML = GEO goldmine. AI models actively look for well-structured Q&A content to cite. A fast, schema-rich FAQ page on your custom backend is one of the highest-value GEO assets you can build.
  • Blog stays on Wix and that's fine. Blog content is still 70-80% visible to AI crawlers on Wix. The blog's value is in fresh, regular content (which AI models weight heavily). The slight rendering disadvantage is offset by the consistency of publishing.
